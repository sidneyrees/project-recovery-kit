## The Project Health Scorecard

| # | Question | Score (1-5) |
|---|---|---|
| 1 | Is scope an endless, growing list with no formal sign-off process? | |
| 2 | Does the team not know what to do next without daily hand-holding? | |
| 3 | Are deadlines constantly moving without anyone re-baselining the plan? | |
| 4 | Is the business side misaligned with what tech is actually building? | |
| 5 | Has the original developer/agency gone unresponsive or slow? | |
| 6 | Is there no single source of truth for current status (specs live in Slack/email/memory)? | |
| 7 | Has a "small fix" repeatedly taken multiples of its original estimate? | |
| 8 | Is there no working staging environment separate from production? | |
| 9 | Does nobody know the current hosting/domain/vendor credentials for certain? | |
| 10 | Has a launch date already been missed at least once? | |
| 11 | Is there no automated backup of the database and codebase? | |
| 12 | Would the business be seriously harmed if the current developer disappeared tomorrow? | |

**Scoring:**
- **12-24: Yellow** — Manageable with process discipline. Start with the Stabilization Plan (Step 3 below).
- **25-42: Orange** — Active recovery needed. Run the full 5-step framework starting now.
- **43-60: Red** — Critical. Freeze scope today, do the 48-hour audit this week, and don't add anything new until it's done.

## Quick Diagnostic (Original 4 Questions)

If you don't have time for the full scorecard, these 4 catch the most common failure patterns:

1. Is scope an endless list? → Scope creep. Freeze scope immediately.
2. Does the team not know what to do next? → Missing PM. I become acting PM for 2 weeks.
3. Are deadlines constantly moving? → Poor estimation. Switch to throughput-based planning.
4. Is business not aligned with tech? → Translation gap. Create a shared glossary.

## My 5-Step Recovery Framework

### Step 1: Technical & Process Audit (48 hours)

**Deliverable:** A written audit covering:
- Codebase health (dependencies, test coverage, tech debt hotspots)
- Infrastructure state (hosting, backups, access/credentials inventory)
- Data integrity (is the database consistent, is there a working backup)
- Team/process state (who knows what, where documentation actually lives)
- A ranked list of the top 5 risks, using the same scoring model as the [Risk Matrix](https://github.com/sidneyrees/pmo-ecommerce-playbook/blob/main/02-risk-matrix.md)

### Step 2: Stakeholder Mapping & Expectation Reset

**Deliverable:** A one-page reset document sent to every stakeholder covering:
- What's actually true about current status (no sugar-coating)
- What changes starting now (single point of contact, new reporting cadence)
- What will NOT change in the next 2 weeks (to reset expectations on speed)

Use the [Weekly Status Report template](https://github.com/sidneyrees/pmo-ecommerce-playbook/blob/main/03-weekly-status-report.md) starting from week one of the reset — consistency rebuilds trust faster than any single announcement.

### Step 3: Stabilization Plan (ship one small thing in 2 weeks)

**Why this matters:** Teams in crisis mode have usually stopped shipping anything. The fastest way to rebuild stakeholder confidence and team morale is a small, real, visible win — not a plan.

**Deliverable:**
- One scoped, shippable item selected specifically because it's achievable in 2 weeks
- A visible "shipped" moment communicated to all stakeholders
- Scope frozen on everything else until this ships

### Step 4: Controlled Execution & Daily Reporting

**Deliverable:**
- Daily 15-minute standups (async is fine if the team is distributed)
- A single tracked backlog — no shadow lists in Slack DMs
- Weekly status reports resume on a fixed cadence

### Step 5: Roadmap to Scalability

**Deliverable:**
- A revised [Project Charter](https://github.com/sidneyrees/pmo-ecommerce-playbook/blob/main/01-project-charter.md) reflecting the real, current state — not the original one that got the project into trouble
- A prioritized backlog with re-baselined estimates
- A recommendation on whether the current vendor/team should continue, be supplemented, or be replaced

## Red Flags Checklist (Screen Before You Even Start the Audit)

Signs a project needs recovery, not just "more time":

- [ ] No one can produce a current, accurate list of what's actually built vs. planned
- [ ] "It's almost done" has been said for more than one full sprint/milestone in a row
- [ ] Access credentials (hosting, domain, payment gateway) are only known to one person, and that person is hard to reach
- [ ] There is no backup, or the last backup is untested
- [ ] The original budget has been exceeded by more than 50% with no formal change order
- [ ] Communication has moved from scheduled meetings to ad-hoc panic threads

## Real Recovery Case Study: Emergency eCommerce Migration & Data Recovery

**Challenge:** A company contacted me during a critical business crisis. Their existing software provider had dramatically increased subscription fees, making the platform financially unsustainable. At the same time, the provider retained control of all business data, and no usable backup was available. With more than 40,000 products, 8,000 recurring customers, and years of operational data locked inside the platform, the company faced the risk of losing access to essential business information and potentially being unable to continue operating.

**Solution:** I designed and executed an accelerated migration strategy focused on speed, data recovery, and business continuity — custom Python bots to extract data, a new WooCommerce infrastructure built in parallel, and validated imports under extreme time pressure.

**Results:**
- Recovered and migrated 40,000+ products and 8,000+ recurring customer records
- Completed the entire migration and platform launch in 7 days
- Restored full ownership and control of all business data
- Reduced annual platform costs to a simple operating model (~10 hours/month support)

**Technologies:** Python · WordPress · WooCommerce · Data Migration · Data Recovery · Automation · Web Scraping · Business Continuity
